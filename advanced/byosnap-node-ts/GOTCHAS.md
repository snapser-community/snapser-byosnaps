# Gotchas

## TSOA Route Generation

1. **Routes are auto-generated**: The `src/routes.ts` file is generated by TSOA. Do not edit it manually. Run `npm run build:routes` or `npm run build` to regenerate.

2. **Controller imports in server.ts**: You MUST import all controllers in `server.ts` for TSOA to register their routes. If you add a new controller file, add an import line.

3. **Decorator order matters**: TSOA decorators should follow this order: `@Route` → `@Get/@Post/@Put/@Delete` → `@Extension` → `@Response` → `@Middlewares`.

4. **x-snapser-auth-types**: Use `@Extension("x-snapser-auth-types", ["user", "api-key", "internal"])` to declare which auth types Snapser should expose in the SDK/API Explorer.

## Auth Middleware

1. **Three auth types**: `user` (session token), `api-key` (server-to-server), `internal` (gateway header for intra-Snapend calls).

2. **Gateway header**: For internal-only endpoints, the `Gateway: internal` header must be present. This is set automatically by the Snapser gateway for service-to-service calls.

3. **User-Id validation**: For `user` auth, the middleware validates that the `User-Id` header matches the `userId` path parameter.

## Swagger Generation

1. **TSOA generates OpenAPI 3.x**: The `tsoa.json` is configured for `specVersion: 3`. The output goes to `snapser-resources/swagger.json`.

2. **fix-swagger.ts**: The post-processing script moves `x-description` to `description` fields and fills in empty response descriptions. This runs automatically as part of `npm run build:swagger`.

## Storage API (snapser-internal SDK)

1. The `snapser-internal/` directory contains auto-generated API client code.
2. In this example, Storage API calls are stubbed with TODO comments.
3. The `go.mod` / `package.json` does NOT include the snapser-internal dependency — add it when ready.

## Blob Ownership Patterns

1. **Config blobs**: Owner key is `{tool_id}_{environment}` (e.g., `characters_dev`). Access type is `private`.
2. **User data blobs**: Owner key is the user ID. Access type is `protected`.
3. **CAS (Compare-And-Swap)**: Always fetch the current CAS before replacing a blob to avoid race conditions.

## Docker

1. The Dockerfile uses a multi-stage build: build stage installs all deps and compiles TypeScript, production stage only copies the compiled output.
2. Port 5003 is exposed by default.
3. The `npm run build` command also generates the swagger spec.
